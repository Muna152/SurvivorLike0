## 1. 总览：文档生命周期地图

```
灵感 → 概念 → 预生产 → 生产 → 打磨 → 发布 → DLC
 │      │       │        │      │      │      │
 │      │       ├─ GDD    │      │      │      │
 │      ├─ 概念文档 ──────┤      │      │      │
 │      │       ├─ TDD    ├─ FS  │      │      │
 │      │       ├─ Art Bible ├─ Asset List ─┤  │
 │      │       ├─ Audio Bible │      │      │  │
 │      │       ├─ 项目计划   ├─ 进度  │      │  │
 │      │       │        │      │      │      │
 ▼      ▼       ▼        ▼      ▼      ▼      ▼
 ─────────────────────────────────────────────────
  维护强度：  创建    持续更新   更新   冻结   更新
```

### 文档两大受众

| 受众 | 特点 | 文档风格 |
|------|------|---------|
| **人类开发者** | 需要上下文、理由、取舍 | 自然语言 + 表格 + 流程图，允许口语化 |
| **AI 智能体** | 需要结构化、可索引、无歧义 | Markdown 结构 + 前置索引 + 交叉链接，避免模糊描述 |

> 同一份信息，人类文档写"为什么"，AI文档写"是什么"。两者共存互补。

## 2. 阶段一：灵感期

**阶段特征**：只有模糊想法，可能是"我想做一个像Vampire Survivors的游戏"。

### 需要的文档

#### 灵感笔记（Idea Notes）

| 项 | 说明 |
|----|------|
| 何时创建 | 灵感出现时，随手记 |
| 颗粒度 | 极粗，关键词即可 |
| 谁维护 | 产生灵感的人 |
| 存放 | 个人笔记 / Notion / 手机备忘录 |

**示例**：

```markdown
# 灵感：暗夜幸存者

- 类型：俯视角生存射击
- 参考VS，但角色有主动技能
- 想法：黑暗中只有角色周围有光，灯油耗尽就死
- 时间：15-30分钟一局
- 差异化：光 vs 暗的核心机制，不只是堆怪
- 可能的问题：光机制会不会太压抑？需要验证
```

> **颗粒度判断**：这个阶段不需要任何正式文档。灵感笔记的目的是"别忘了"，不是"说清楚"。

## 3. 阶段二：概念期

**阶段特征**：灵感经过初步验证，决定"要做这个游戏"。需要统一团队认知。

### 3.1 概念文档（Concept Document / One-Pager）

| 项 | 说明 |
|----|------|
| 何时创建 | 决定立项时 |
| 颗粒度 | 1页，5分钟读完 |
| 维护频率 | 立项后基本冻结，大幅转向时才改 |
| 受众 | 全团队 + 投资方/发行商 |

**示例**：

```markdown
# 概念文档：暗夜幸存者

## 一句话
俯视角黑暗生存游戏，在有限光源下对抗无尽敌群，用光与武器活过30分钟。

## 是什么
- 俯视角 2D 生存射击（类 Vampire Survivors）
- 自动攻击 + 手动移动 + 主动技能
- 核心：光/暗机制——角色周围有光照范围，灯油耗尽则视野缩小至危险程度
- 单局 15-30 分钟

## 给谁玩
- Vampire Survivors / Brotato 玩家
- 喜欢数值成长但不想深度操作的休闲玩家

## 为什么不同
- 市面类VS游戏缺少环境压力机制，光/暗系统提供额外紧迫感
- 主动技能增加操作深度，不是纯挂机

## 大致规模
- 5个可玩角色，15+武器，3个关卡
- 预计开发周期：6-8个月（3人团队）

## 核心风险
- 光/暗机制是否真的好玩？→ 2周内做原型验证
- 同屏大量敌人性能 → 使用对象池 + DOTS评估
```

### 3.2 原型验证记录

| 项 | 说明 |
|----|------|
| 何时创建 | 概念文档之后、正式启动之前 |
| 颗粒度 | 结论 + 关键数据，不需要过程 |
| 维护频率 | 一次性的，验证完即归档 |

**示例**：

```markdown
# 原型验证：光/暗机制

## 验证目标
光/暗机制是否提供有趣的玩法压力？

## 结论：✅ 可行，但需调整

## 关键发现
- 灯油耗尽 → 全黑的感觉比预想更压抑，需要保留最低可视范围
- 灯油作为资源 → 与HP形成双资源管理，玩家反馈正面
- 问题：灯油恢复手段太少，需要增加光源道具

## 数据
- 5名测试者，4人认为"增加紧张感"，1人认为"只是烦"
- 平均存活时间：无灯油限制 28min → 有灯油限制 12min（需平衡）

## 决策
- 保留灯油机制，但增加：最低可视范围、灯油掉落、光源类道具
```

## 4. 阶段三：预生产期

**阶段特征**：验证核心玩法可行，开始全面设计。这是文档创建最密集的阶段。

### 文档创建顺序与依赖

```
GDD ←── 概念文档推导
 │
 ├── TDD ←── GDD的技术需求推导
 │
 ├── Art Bible ←── GDD的视觉需求推导
 │
 ├── Audio Bible ←── GDD的听觉需求推导
 │
 └── 项目计划 ←── GDD的工作量估算推导
```

---

### 4.1 游戏设计文档（GDD）

| 项 | 说明 |
|----|------|
| 何时创建 | 预生产期开始 |
| 颗粒度 | 中等——定义"做什么"，不定义"怎么做" |
| 维护频率 | 预生产期高频更新；生产期仅重大设计变更时更新 |
| 受众 | 策划（核对设计）、程序（理解需求）、美术（理解视觉） |

**颗粒度标准**：
- ✅ 写：核心循环、系统规则、数值公式、状态定义、边界条件
- ❌ 不写：具体代码实现、UI像素级布局、逐帧动画描述

**示例（核心循环部分）**：

```markdown
## 核心游戏循环

### 单局流程
主菜单 → 角色选择 → 进入关卡 → 战斗生存(30min) → 胜利/死亡 → 结算 → 返回

### 局内循环
1. 移动躲避敌人
2. 武器自动攻击
3. 击杀 → 掉落经验宝石
4. 拾取 → 累积升级
5. 升级暂停 → 3选1（新武器/武器升级/被动道具）
6. 角色变强 → 面对更强敌人 → 回到1

### 局外循环
1. 结算 → 检查解锁条件 → 可能解锁新角色
2. 金币持久化 → 永久升级商店
3. 选择角色+永久加成 → 开始下一局

### 关键参数
- 单局时长：30分钟
- 最高等级：30
- 升级经验公式：`5 + 3 × 等级²`
- 同屏敌人上限：500
```

### 4.2 技术设计文档（TDD）

| 项 | 说明 |
|----|------|
| 何时创建 | GDD核心循环定稿后 |
| 颗粒度 | 数据结构 + 接口签名 + 关键决策理由，不写实现代码 |
| 维护频率 | 系统设计变更时同步更新 |
| 受众 | 程序员 |

**颗粒度标准**：
- ✅ 写：架构选型+理由、核心类/接口定义、数据结构、状态机、性能指标、风险项
- ❌ 不写：方法体实现、局部变量、算法细节（除非是核心难点）

**示例（对象池系统部分）**：

```markdown
## 对象池系统

### 设计决策
- 使用泛型对象池而非ECS：项目规模小，ECS学习成本高，对象池足以满足500+同屏需求
- 池预预热（Prewarm）：在场景加载时预创建对象，避免运行时卡顿

### 核心接口
```csharp
public class ObjectPool<T> where T : Component
{
    T Get();                    // 从池获取，池空则创建
    void Return(T obj);         // 归还池中
    void Prewarm(int count);    // 预创建
}
```

### 关键约束
- 所有频繁创建/销毁的对象（敌人、投射物、掉落物、特效）必须走对象池
- 严禁运行时 Instantiate/Destroy 这些对象
- 池上限：敌人300、投射物200、掉落物500、特效100

### 性能指标
- Get/Return 操作 < 0.1ms
- Prewarm 300个敌人 < 3帧
```

### 4.3 美术风格指南（Art Bible）

| 项 | 说明 |
|----|------|
| 何时创建 | 预生产期，与GDD并行 |
| 颗粒度 | "宪法级"——定规则和禁区，不定具体像素 |
| 维护频率 | 定稿后极少修改，仅风格大转向时 |
| 受众 | 美术、外包、程序（理解视觉需求） |

**示例**：

```markdown
## 色彩体系

### 主色调
- 暗蓝 #1a1a2e — 背景/夜空
- 暗紫 #16213e — 深层阴影
- 橙金 #e2b714 — 强调色/光源/掉落物
- 血红 #c0392b — 受击/危险/HP

### 色彩禁区
- ❌ 不用纯白 #ffffff 做大面积填充（只在光源中心小范围使用）
- ❌ 不用高饱和绿色（与掉落物经验宝石的绿色冲突）
- ❌ 背景色相不进暖色域（暖色留给互动元素）

### 角色比例
- 3头身（Q版偏向）
- 轮廓线：2px黑色描边
- 动画帧率：8帧/秒（行走）4帧/秒（待机）

### 参考与反例
- ✅ 参考：Hollow Knight 的暗色氛围 + Castlevania 的哥特元素
- ❌ 反例：不要像 Dead Cells 那样的高对比霓虹风格
```

### 4.5 项目计划（Project Plan）

| 项 | 说明 |
|----|------|
| 何时创建 | 预生产期末尾，所有设计文档就绪后 |
| 颗粒度 | 里程碑级，精确到周，不精确到天 |
| 维护频率 | 每2周回顾，每里程碑更新 |

**示例**：

```markdown
## 里程碑计划

| 里程碑 | 目标 | 周数 | 交付物 |
|--------|------|------|--------|
| M0: 原型 | 验证核心玩法 | 2 | 可玩原型 |
| M1: MVP | 核心循环可玩 | 3 | 1角色+1武器+1敌人 |
| M2: 系统完善 | 武器/升级/掉落全通 | 3 | 完整单局体验 |

## 5. 阶段四：生产期

**阶段特征**：设计定型，开始大量制作内容。文档从"创建"转向"维护"和"细化"。

### 5.1 功能规格书（Feature Spec / FS）

| 项 | 说明 |
|----|------|
| 何时创建 | 每个功能模块开发前 |
| 颗粒度 | 中细——足以让另一个程序员独立实现 |
| 维护频率 | 开发中迭代，完成后归档 |
| 受众 | 程序（实现依据）、QA（测试依据）、策划（需求确认） |

**颗粒度标准**：
- ✅ 写：功能目标、交互流程、状态定义、数据结构、边界条件、异常处理
- ❌ 不写：具体实现逻辑、代码实现

**示例（升级系统FS）**：

```markdown
## 升级系统 - 功能规格书

### 目标
玩家累积经验后暂停游戏，从3个随机选项中选择强化。

### 交互流程
1. 经验值 >= 升级所需 → 触发升级
2. 游戏暂停（Time.timeScale = 0）
3. 随机从候选池中抽取3个不重复选项
4. 显示升级UI：3张卡片，每张显示名称+描述+图标
5. 玩家点击选择 → 应用效果 → 关闭UI → 游戏恢复

### 选项类型
| 类型 | 来源 | 示例 |
|------|------|------|
| 新武器 | 武器池（尚未拥有的） | "获得飞剑" |
| 武器升级 | 已拥有武器的升级路径 | "飞剑 → 双重飞剑" |
| 被动道具 | 被动池 | "移速+10%" |

### 选项权重
- 新武器：权重 30（已有武器越少权重越高）
- 武器升级：权重 40
- 被动道具：权重 30
- 幸运值每点增加高品质选项权重 +5%

### 边界条件
- 所有武器已获得 → 新武器选项不再出现，权重重分配
- 武器已满级 → 该武器升级不再出现
- 武器槽位已满（6个）→ 新武器选项不再出现
- 升级期间受伤 → 不计算（游戏已暂停）

### 数据结构
- UpgradePool：所有可用选项的ScriptableObject列表
- UpgradeOption：{ 类型, 关联数据, 权重, 前置条件, 最大持有数 }
- PlayerUpgrades：当前已选升级的运行时记录

### 异常处理
- 候选池为空（极端情况所有选项都用完）→ 显示"全满"，直接恢复游戏
- 3个选项不够（候选池<3）→ 有几个显示几个
```

### 5.3 技术设计文档（TDD）— 细化版

| 项 | 说明 |
|----|------|
| 何时创建 | 每个系统开发前，基于GDD写TDD |
| 颗粒度 | 数据结构定义 + 接口签名 + 状态机 + 关键决策理由 |
| 维护频率 | 实现过程中发现设计需调整时更新 |

> 生产期的TDD比预生产期更细。预生产期定义架构方向，生产期定义具体接口。

**示例（升级系统TDD部分）**：

```markdown
## 升级系统 - 技术设计

### 架构
- UpgradeManager（MonoBehaviour单例）：管理升级流程
- UpgradePool（ScriptableObject）：配置数据，策划可编辑
- PlayerUpgrades（运行时数据）：当前已选升级记录
- UpgradeUI（MonoBehaviour）：UI表现

### 核心接口
```csharp
// UpgradeManager
void TriggerLevelUp();
UpgradeOption[] RollOptions(int count);  // 从池中抽取N个选项
void ApplyOption(UpgradeOption option);  // 应用选择的升级

// UpgradePool
List<UpgradeOption> GetAvailableOptions(PlayerUpgrades current); // 过滤已满选项

// PlayerUpgrades
bool HasUpgrade(string upgradeId);
int GetUpgradeLevel(string upgradeId);
void AddUpgrade(UpgradeOption option);
```

### 关键决策
- 升级暂停用 Time.timeScale = 0，不用单独的暂停状态：简化状态机
- 选项随机用加权随机（WeightedRandom工具类），不用洗牌：需要权重支持
- 升级记录用 List<UpgradeRecord>，不用 Dictionary：保留选择顺序，方便调试

### 风险
- 后期选项多时 RollOptions 性能 → 当前池<100项，不构成风险
- 保存/读档时升级状态恢复 → UpgradeRecord 需要可序列化
```

## 10. AI可读文档体系：docs/结构化目录

### 10.1 核心理念

来自 OpenAI Codex 团队的实践经验：

> **给AI一张地图，不是一本1000页的说明书。**

关键原则：
1. **渐进式披露**：AI从小入口开始，按需深入，不被信息淹没
2. **代码仓库即真相**：存储在代码仓库外的信息对AI等于不存在
3. **结构化+可索引**：文件名和目录结构本身就是导航系统
4. **交叉链接**：文档之间必须互相引用，形成知识网络
5. **机械验证**：用linter/CI检查文档的完整性、新鲜度、交叉引用
6. **按需加载**：AI只读当前任务相关的文件，杜绝一次性加载全部文档

### 10.2 旧结构的问题与修正

#### 问题1：GDD/TDD 单文件膨胀

**旧结构**：
```
Docs/
├── GDD.md          # ← 3000+行，AI读完就没context了
├── TDD.md          # ← 2000+行，大部分与当前任务无关
```

**问题**：AI要修改武器系统，被迫读取包含角色、敌人、UI等全部内容的GDD，浪费大量context。

**修正**：按系统拆分为独立文件，每个文件控制在200行以内。AI只读需要的那个文件。

#### 问题2：GDD章节 vs feature-specs 职责模糊

| 疑问 | 回答 |
|------|------|
| GDD/weapon-system.md 和 feature-specs/weapon-system.md 重复吗？ | **不重复，职责不同** |
| GDD写什么？ | **设计意图**：武器有什么类型、怎么升级、数值公式（做什么+为什么） |
| feature-specs写什么？ | **实现契约**：交互流程、状态机、接口签名、边界条件、异常处理（怎么做） |
| 为什么不分一个文件？ | 读GDD的人（策划/美术）不需要看接口签名；写代码的AI不需要读设计背景 |

**一句话区分**：GDD是"玩家体验到什么"，feature-spec是"程序员怎么实现"。

#### 关键推论：哪些系统需要 feature-spec？

| 系统类型 | tdd/ | feature-specs/ | 原因 |
|----------|------|----------------|------|
| 基础设施（对象池、事件总线、存档） | ✅ 必须有 | ❌ 不需要 | 没有用户交互流程，实现者看TDD就能写 |
| Gameplay系统（升级、武器、敌人生成） | ✅ 必须有 | ✅ 必须有 | 有交互流程、状态机、边界条件，TDD管不住这些 |

判断标准很简单：**这个系统有"用户经历什么流程"吗？** 如果有，写feature-spec；如果没有，TDD就够了。

#### 问题3：ROADMAP vs TASKS 功能重叠

**旧结构**：
```
Docs/
├── ROADMAP.md      # 里程碑级任务列表
├── TASKS.md        # 同一批任务的原子拆解
```

**问题**：ROADMAP写"实现升级系统 ✅"，TASKS也写"升级系统 → T2.3.1 实现UpgradeManager ✅"。同一信息维护两处，AI也分不清该看哪个。

**修正**：
- ROADMAP 只保留**里程碑+阶段概览**（战略层，给人看进度）
- 具体任务拆解归入 `exec-plans/active/`（执行层，给AI看具体步骤）
- TASKS.md 删除，其内容分散到各exec-plan中

#### 问题4：design-docs/ 定位不清

`core-beliefs.md` 其实是"项目宪法"，应提升到根目录，与AGENTS.md同级，让AI第一时间读到。不需要单独一个design-docs/目录。

### 10.3 修正后的推荐目录结构

```

### 10.4 文件大小约束

| 文件 | 建议上限 | 超了怎么办 |
|------|---------|-----------|
| AGENTS.md | 100行 | 只保留导航和硬约束，详情移到对应文件 |
| CORE_BELIEFS.md | 50行 | 只保留最核心的5-7条原则 |
| ARCHITECTURE.md | 150行 | 只保留架构图和管理器关系，细节移到tdd/ |
| gdd/*.md | 200行/文件 | 一个系统超过200行，考虑拆分子系统 |
| tdd/*.md | 200行/文件 | 同上 |
| feature-specs/*.md | 300行/文件 | 可以稍长，因为AI实现时需要完整契约 |
| exec-plans/active/*.md | 150行/文件 | 超过说明任务太大，应拆分为多个exec-plan |
| ROADMAP.md | 100行 | 只有里程碑，不拆任务 |

### 10.5 AI读取路径：按任务类型决定读什么

AI最常见的几种任务，每种只需要读取特定文件，避免加载无关内容：

#### 任务：新增一个武器

```
1. AGENTS.md              → 知道硬约束（命名规范、对象池等）
2. gdd/weapons.md         → 理解武器设计（类型、数值、升级树）
3. tdd/object-pool.md     → 投射物必须走对象池
4. feature-specs/weapon-system.md → 实现契约（接口、数据结构、流程）
5. CORE_BELIEFS.md        → 不需要读（武器设计不涉及核心原则变更）
```

#### 任务：修改升级选项权重

```
1. AGENTS.md              → 知道数值不硬编码的约束
2. gdd/upgrades.md        → 当前权重设计和设计意图
3. feature-specs/upgrade-system.md → RollOptions的实现逻辑
4. 不需要读tdd/、ROADMAP、其他gdd/
```

#### 任务：修BUG（升级时卡顿）

```
1. AGENTS.md              → 性能约束
2. feature-specs/upgrade-system.md → 升级流程，定位卡顿位置
3. tdd/event-system.md    → 如果卡顿与事件有关
4. exec-plans/tech-debt-tracker.md → 看是否已有相关技术债务
```

#### 任务：新增一个系统（成就系统）

```
1. AGENTS.md              → 硬约束
2. CORE_BELIEFS.md        → 确认不违反核心原则
3. ARCHITECTURE.md        → 新系统在架构中的位置
4. gdd/index.md           → 确认成就系统不在已有GDD中
5. → 先写 gdd/achievements.md → 再写 tdd/achievements.md → 再写 feature-specs/achievements.md
```

### 10.6 三层文档的关系：GDD → TDD → Feature-Spec

```
GDD（设计意图）          TDD（技术方案）         Feature-Spec（实现契约）
"武器自动攻击最近敌人"  → "WeaponBase抽象基类"  → "FindNearestEnemy()调用逻辑"
"最多6个武器槽"         → "PlayerWeapons.List"  → "槽位满时升级选项过滤规则"
"升级3选1"              → "UpgradeManager单例"   → "RollOptions(3)的完整交互流程"

    读者：策划/美术         读者：主程/架构师         读者：实现者（人或AI）
    问题：做什么？为什么？    问题：怎么组织？什么结构？  问题：怎么实现？边界在哪？
```

**去重规则**：
- 同一条信息只在一个层级写完整，其他层级只引用
- GDD写了"升级经验公式：5+3×L²" → TDD不重复，只写"经验计算见gdd/upgrades.md"
- TDD写了"ObjectPool<T>.Get()"接口 → feature-spec不重复，只写"使用PoolManager.Get()"
- feature-spec写了完整的交互流程 → GDD只写设计意图，不写流程细节

### 10.7 AGENTS.md — AI唯一入口

### 10.11 CORE_BELIEFS.md — 项目宪法

与AGENTS.md同级，定义**不可违反的设计原则**。AI入口文件直接引用它。

**示例**：

```markdown
# 核心设计信念

> 不可违反。新功能违反以下任一条，必须经全团队讨论。

1. **单局25-35分钟** — 不做无限模式，机制导致局时偏移>5分钟则必须调整
2. **操作极简，深度在策略** — 只需移动，攻击全自动，主动技能上限2个
3. **每次选择都有意义** — 无"必选"（>80%选取率则削弱）无"废选项"（<5%则加强）
4. **性能是功能** — 60FPS/500敌人，新功能降帧>10%视为BUG，对象池强制
5. **数值不硬编码** — 策划可调数值一律ScriptableObject
```

### 10.14 人类文档 vs AI文档 对比

| 维度 | 人类文档 | AI文档 |
|------|---------|--------|
| 入口 | 文件夹浏览、口头询问 | `AGENTS.md`（唯一入口） |
| 结构 | 线性长文可以 | 拆分短文件 + index导航，渐进披露 |
| 导航 | 目录页、搜索 | 文件头部链接表 + index.md |
| 颗粒度 | 允许模糊描述和口语化 | 必须精确、无歧义、可机械检查 |
| 理由 | 必须写"为什么" | 必须写"为什么"（AI也需要决策上下文） |
| 维护 | 靠人的纪律 | 靠linter + CI + doc-gardening智能体 |
| 状态 | 隐含（"这文档还在用吗？"） | 显式标记（状态+最后更新日期） |
| 废弃 | 慢慢没人看了 | 标记`[已废弃]`+日期+重定向链接 |
| 存储 | 可能散落在Confluence/飞书 | **必须在代码仓库内**（仓库外对AI不存在） |
| 读取方式 | 可以一口气读完GDD | **只读当前任务相关的1-3个文件** |

### 10.15 AI文档维护实践

#### 文档健康检查（推荐CI化）

```markdown
## 文档健康检查清单（可脚本化）

- [ ] AGENTS.md 行数 < 100
- [ ] CORE_BELIEFS.md 行数 < 50
- [ ] gdd/ tdd/ feature-specs/ 每个文件 < 300行

```

#### Doc Gardening（文档园丁）

定期（建议每周）由人或AI智能体执行：

1. 扫描所有文档，检查交叉链接是否有效
2. 标记超过1个月未更新的活跃文档（可能已过时）
3. 检查"已完成"功能的FS是否与代码实际行为一致
4. 检查gdd/tdd/feature-specs是否有内容重复（违反去重规则）
5. 清理已废弃文档或添加重定向

## 11. 附录：颗粒度速查表

### 按文档类型的颗粒度

| 文档 | 太粗（❌） | 刚好（✅） | 太细（❌） |
|------|----------|----------|----------|
| 概念文档 | "做一个生存游戏" | 1页：类型+差异化+目标受众+核心风险 | 3页以上的详细玩法描述 |
| GDD（单系统文件） | "角色有属性" | 属性名+数值+公式+边界条件 | 逐帧动画描述 |
| TDD（单系统文件） | "用对象池" | 接口签名+数据结构+关键决策+理由 | 具体实现代码 |
| FS | "做一个升级系统" | 交互流程+状态定义+边界条件+数据结构 | 代码级伪实现 |
| Asset List | "需要一些UI" | 每条：ID+类型+规格+状态+负责人 | 每条含详细的美术指导 |
| Art Bible | "风格偏暗" | 色彩HEX+参考图+禁区+比例规范 | 逐像素的色值表 |
| 执行计划 | "加一个Boss" | Step级拆解+验收标准+决策记录 | 代码级步骤描述 |
| AGENTS.md | 完整的GDD内容 | 纯导航+硬约束（<100行） | 任何详细信息 |
| index.md | 只有文件名 | 系统+一句话概要+链接+状态 | 重复文档内容 |

### 按团队规模的文档精简建议

| 文档 | 1-3人 | 4-10人 | 11-20人 |
|------|-------|--------|---------|
| 概念文档 | ✅ 必须有 | ✅ 必须有 | ✅ 必须有 |
| GDD | 单文件精简版 | 按系统拆分到gdd/目录 | 按系统拆分+每个系统有详细数值 |
| TDD | 代码注释即TDD | 按系统拆分到tdd/目录 | 每个系统一份TDD |

### AI Context预算参考

假设AI可用context约128K tokens（约10万汉字），建议单次任务文档读取量：

| 场景 | 读取文件数 | 预估tokens | 说明 |
|------|-----------|-----------|------|
| 小修改（改数值） | 1-2个 | <3K | gdd/对应系统.md + ScriptableObject |
| 新增功能 | 3-4个 | 5-10K | AGENTS.md + gdd/ + tdd/ + feature-specs/ |
| 修BUG | 2-3个 | 3-8K | feature-specs/ + tdd/ + tech-debt |
| 新增系统 | 4-5个 | 10-15K | AGENTS.md + CORE_BELIEFS + ARCHITECTURE + gdd/ + tdd/ |
| 大重构 | 5-6个 | 15-25K | 上述全部 + exec-plans/ |

> **原则**：任何单次任务，AI读取的文档总量不应超过context的25%。剩余空间留给代码理解、对话历史和生成输出。

## 结语

> **好的文档体系像好的代码架构：低耦合、高内聚、依赖方向清晰。**
>
> 人类文档解决"为什么"，AI文档解决"是什么"和"在哪里"。
> 两者共享同一份信息源，但组织方式不同。
>
> 最终检验标准：**一个新加入的人（或AI），能在不问任何人的情况下，找到完成工作所需的全部信息。**
